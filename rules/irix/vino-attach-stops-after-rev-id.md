# VINO attach stops after REV_ID read — IRIX kernel never registers a board

**Keywords:** vino, indycam, vid, vino_init, REV_ID, probe, no boards found, videod, VID_GETBOARDINFO
**Category:** vino, irix
**Status:** Open. Diagnosed in IRIX 5.3 for Indy and 6.5.22 installs; root cause sits inside the IRIX `vino_init()` driver path on iris but the failure is *silent* — no SYSLOG or console message, no further MMIO activity after a single REV_ID read.

## Symptom

After a clean IRIX install with the right packages (`vino.sw.eoe`, `vl_eoe`,
`dmedia_eoe.sw.base`, etc.), `/usr/etc/videod` fails immediately:

    IRIS # /usr/etc/videod

    Fatal server error:
    no boards found

`vlinfo` then says `can't open server ''`. `hinv` confirms the chip is
visible (`Vino video: unit 0, revision 0`), so the kernel knows vino
exists at the hardware-inventory level, but the vid driver has zero
registered boards.

## What iris emits at the bus level

Instrument `Vino::read_reg` with `eprintln!`. Across an entire `iris
--ci` lifetime (PROM POST → maintenance menu → IRIX boot → login →
`/usr/etc/videod`), **exactly one VINO bus access** is observed:

    VINO READ [0x0000] REV_ID -> 0x000000b0

and nothing else. No writes, no 64-bit reads, no byte/halfword
accesses. The single read happens *after* PROM POST (PROM phase ends
with zero vino reads) — it is the IRIX vino driver, not the PROM POST.

## What that tells us

The IRIX kernel calls `vino_init()` at boot (verified: `master.c`
generated by `autoconfig` lists `vino_init` in the init-function
array, and the kernel `/unix` strings include the driver's printf
literals like `"Vino: missing"`, `"Vino: not vino!"`, and `"Vino
Video, Rev: %d"`). `vino_init` reads `REV_ID`, sees `chip_id = 0xB`
(correct — iris returns `0xB0`), and would normally print `"Vino
Video, Rev: 0"` on success or one of the `"Vino: …"` warnings on
failure. **None of those messages appear**, so the driver hasn't
classified the chip as either present-ok or bogus — it has aborted
before reaching either branch.

The single REV_ID read + no follow-up + no diagnostic message means
some condition *after* the REV_ID read but *before* any further chip
interaction is making `vino_init` return early. Candidates:

1. **Memory/DMA arena not allocated.** The string `"Vino: DAP table
   needs %d descriptors, only have %d."` and `"Vino: Cannot allocate
   DAP table for %d descriptors."` are in the kernel — could fire here
   but we don't see them. Unlikely if dmrb is up.
2. **vid clone driver registration glue.** vino registers as a
   sub-driver of vid (`master.d/vino` says `s vino_ - - vid`). If the
   call into `vid_register_board()` (or equivalent) fails, vino
   wouldn't go further. iris's emulated vid sub-system might not be
   advertising the right "I am ready" state.
3. **IRQ vector wiring on the GIO bus.** vino raises interrupts via
   the HPC3/IOC chain. If the irq object iris hands to vino isn't
   matching what the IRIX driver expects to find (e.g., wrong
   interrupt number), `vino_init` could bail.
4. **`mtune` parameters silently rejected.** `master.c` declares
   `vino_mtune_dmrbpages = 0x0; vino_mtune_flags = 0x0;` — both
   zero. The driver may treat 0 as "disabled" and skip
   registration. The `vino` mtune file ships with defaults (see
   `/var/sysgen/mtune/vino`) but they may not be applied.

The single REV_ID read also gives a forced re-read of `chip_id` —
changing iris's `RESET_VAL` from `0xB0` to `0xB1` (chip_id still
`0xB`, rev `1`) was tested and changed nothing: still one read, still
no boards.

## How to verify the PROM is not the source of the read

Disprove the PROM-POST theory like this:

    # Add eprintln! to Vino::read_reg in src/vino.rs (after the dlog_dev! line).
    # Build, then:
    rm -f /tmp/iris.sock
    ./target/release/iris --ci > /tmp/iris-probe.log 2>&1 &
    ./target/release/iris-ci start
    # Wait until "Option?" appears in irix-install-console.log:
    until tail -c 200 irix-install-console.log | tr -d '\r' | grep -q 'Option?'; do sleep 2; done
    grep -c '^VINO' /tmp/iris-probe.log     # expect 0 — PROM has not touched vino

After picking `1` to Start System and entering `boot` at sash, the
count rises to exactly 1 (the IRIX kernel's `vino_init` REV_ID read).

## What to try next

Each of these is its own investigation; none has been attempted.

- **Disassemble `vino_init` in `/unix`.** `dis /unix vino_init`
  (IRIX dis(1)) or `objdump -d /unix | sed -n '/<vino_init>:/,$p'` on
  a host with a MIPS objdump. Find the conditional branch right after
  the REV_ID read — that's the silent abort.
- **Instrument iris's vid emulation** (the vid clone-driver
  registration surface) — there isn't really one today; vino expects
  to call `vid_register_board()` but iris doesn't intercept that
  (the call lives entirely inside the guest kernel). Confirm what
  *guest-side* state vino expects vid to publish (e.g., the
  `VidBoards` array slot count from `master.d/vid`'s `VidBoardLimit =
  8`) so we can verify it's there at the time `vino_init` runs.
- **Check IRQ wiring.** Look at how iris's vino raises interrupts
  vs what the IRIX driver expects to register against. If the
  irq cell iris hands to vino doesn't have an `intr` table the
  driver can hook, vino bails.
- **Disable the JIT** to rule out a hot-loop optimization caching
  the REV_ID value and suppressing subsequent reads (unlikely for
  uncached KSEG1 MMIO, but worth elimination).

## Useful crumbs

- `src/vino.rs`: `VINO_BASE = 0x00080000`, `REV_ID @ 0x0000`,
  `RESET_VAL = 0xB0` (chip_id=0xB, rev=0).
- `src/physical.rs`: `vino_gio_alias` (`AliasBus` offset
  `0xE1000000`) routes GIO64-aperture access at `0x1F080000` back to
  the primary `0x00080000` mapping. Verified end-to-end correct via
  monitor `vino status`.
- `/var/sysgen/master.d/vino` (in the installed IRIX 5.3 root):
  `s vino_ - - vid` — software driver, depends on vid.
- `/var/sysgen/master.d/vid`: `soc vid 203 - clone` — char major 203,
  clone semantics.
- `/var/sysgen/master.c` (autoconfig-generated): includes
  `vino_init,` in the init-function array and links `vino.a` ahead of
  `vidstubs.a`, so the real driver wins at link time.
- Kernel `/unix` strings include `"Vino: missing"`, `"Vino: not
  vino!"`, `"Vino Video, Rev: %d"`, and `"!<1>IndyCam not attached.
  [HELP=VINONOCAMERA_WARN]"` — none of which we ever see.
